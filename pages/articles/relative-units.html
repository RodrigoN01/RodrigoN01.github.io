Relative Units

Pixels

In this article I will be covering a topic that many of us find complicated at first and prefer to postpone its learning, but trust me, it is worth investing your time on this topic, the relative units.

HTML has been around for 26 years, and pixels have been around as a layout measuring unit since then. Pixels did not came out with the web, but the pixels we use in the web are not exactly the same pixels we knew from before. With that in mind, I thing we should question our trust on them.

Pixels are not absolute

Not particularly the ones on the web. In fact, their size vary based on multiple factors: screen resolution, browser renderer, and device renderer, '1px' is not the same distance every time your UI loads. Knowing that: how far can we rely on an "absolute unit" that is not absolute?

Font based units

The containers and boxes are sized relatively to the text. We want our content to be placed surrounded by the boxes and borders we set.

Luckily, CSS provides us with some good tools to solve this problem, I will only mention the 3 main ones:

  • ch: a typographic measure based on the length of the character 0 (zero).

  • em: a typographic measure based on the length of the character 'm'.

  • rem: a web measure based on the 'em' of the root element.

'em' vs 'ch'

Now, we need to choose which one gets the job done in a more and consistent way. In order to choose, it's required first to understand how each of them are defined.

In CSS, the 'em' is a general unit of length related directly to font size. We use it to define widths, heights, padding, margins and other measurements in both horizontal and vertical directions â€“ something typographers with a traditional background may find surprising. The 'em' unit can also be used to define the font size itself.

The 'ch' unit can be useful if you want to set lengths or sizes which relate directly to the width of the font.

The 'm' and its width

Probably the first question your are asking yourself is, "Why the 'm' ?" When creating a new typeface, designers use (or used) a squared block (kerning) which served as reference to assure each character was proportionally designed in comparison to the other ones. The 'm' is the only character in our entire alphabet that would touch the left and right boundaries of this box. So, '1em' is actually the maximal width of a character, regardless of which font-family it belongs to.

Absolute zero

The number zero is normally the average width of all characters (alphanumeric) in a typeface. The 'ch' is, therefore, an estimation of the character length for a specific font, hence its name (character).

Because the zero is an average, it becomes a more reliable unit than 'em'. On the event of a font-family changing for the content, the '5ch' will continue to be the length of 5 characters, whilst '1em/1rem' may change for more or less than before. In a nutshell, '1ch' in Times New Roman displays a different distance than '1ch' in 'Helvetica' which would be also different than in a monospace typeface.

Using percentages

The use of percentages is very common and powerful, but can also be tricky. Depending on the CSS property which you're adding a value to, the percentage relates to a different measure. For padding and margin, for example, the percentage relates to the width of the containing box (regardless if it is vertical or horizontal padding/margin). 


.wrapper {
  width: 10rem; 
  
  height: 3rem;
} 

.child { 
  padding-top: 10%; /* 1rem */
  
  margin-top: 10%; /* 1rem */
}

Also, if for example you're intending to set a percentage value to width, height, and its derivatives, the fraction will correspond to its container, so height: 50% is half the height of the elements container (given it has a measurable length at parsing time). 


.parent {
  width: 40rem;

  height: 20rem;
}

.child {
  width: 50%; /* 20rem */

  height: 70%; /* 14rem */
}

To some extent, I like to think '%' as the 'em' equivalent of viewport sizes. They are both extremely powerful, but they also require a level of understanding on how CSS rendering will work and a more strategic view of how the UI elements will integrate and interact with each other because they depend on the context where they are inserted.

Viewport based units

Viewports, on the other hand, are much more intuitive. It is part of CSS3, so they are quite new, though their learning curve helps a lot on advocating towards their usage.

  • vh: Viewport height. 1vh is 1% of the actual viewport's height.

  • vw: Viewport width. 1vw is 1% of the actual viewport's width.

  • vmax: Viewport max. 1vmax is 1% of the viewport's biggest dimension

  • vmin: Viewport min. 1vmin is 1% of the viewport's smallest dimension

As you may have come to realise, the viewport units do not depend on the context they are inserted into. So, to keep the comparison between units: if '%' is the 'em' of viewport, viewport units are the 'rem' .

So what should we use?

It's not actually a question of what to use, but more a question of when should we use which.

In other words, we should try to find the message on the layout we are creating, once we do that, the best unit will reveal itself as the one which is more likely to provide a reliable measure of it.

Let's go through a quick example, if you have a piece of display text-base (e.g.: a blog post), you can narrow it down your options to 'rem' , 'em' , or 'ch' . Now, does the specific element in question depends on its parent? (if yes, 'em' or 'ch' probably). Do you need it to be scalable in case you switch typefaces? (if yes, definitely 'ch' ). Of course, this situation is oversimplified, but I believe it would provide a good understanding of the topic

Remember, there is no silver bullet and do not be shy of using every resource at your disposal.
